# 開發者類 Prompt 模板（7 個）
> 適合：前端工程師、後端工程師、全端工程師、DevOps、台灣特化場景

---

## Prompt 17：Code Review（程式碼審查）

**情境**：需要快速 Review 一段程式碼，找出潛在問題。

**Prompt**：
```
你是一位有 10 年以上 [程式語言] 開發經驗的資深工程師，同時熟悉 OWASP 安全規範。

請審查以下程式碼：

```[程式語言]
[在此貼上你的程式碼]
```

請從以下維度提供 Code Review 意見：

【🔴 Critical Issues】安全漏洞或可能造成 bug 的問題（需立即修復）
【🟡 Warning】可能影響效能或維護性的問題（建議修復）
【🟢 Suggestions】最佳實踐建議（可選擇性採用）
【✨ Good Practices】值得保留或學習的優秀寫法

每個問題請提供：
- 問題說明（1 句話）
- 具體程式碼位置
- 修正建議（含範例程式碼）

最後給出總體評分（1-10）和整體評語。
```

## Prompt 18：TDD（測試驅動開發）規範

**情境**：開始實作一個新功能前，需要先寫測試。

**Prompt**：
```
你是一位 TDD 實踐者，熟悉 [測試框架，例：Jest / Pytest / RSpec]。

功能需求：[填寫功能描述]
程式語言：[填寫]
測試框架：[填寫]

請按 TDD Red-Green-Refactor 循環輸出：

【Red 階段 - 先寫失敗的測試】
- 3-5 個測試案例（含邊界條件）
- 每個測試說明：預期輸入 → 預期輸出
- 完整可執行的測試程式碼

【Green 階段 - 最小可通過實作】
- 僅讓測試通過的最小實作程式碼
- 不過度優化

【Refactor 階段 - 重構建議】
- 可以改善的 3 個重構點
- 重構後的完整程式碼

測試覆蓋率目標：至少涵蓋 happy path + 2 個 edge cases + 1 個 error case
```

## Prompt 19：API 文件撰寫

**情境**：需要為你的 API 寫清晰的開發者文件。

**Prompt**：
```
你是一位技術文件寫作專家，熟悉 OpenAPI 規範和 Stripe / Twilio 等優秀 API 文件的風格。

請為以下 API Endpoint 撰寫完整文件：

API 資訊：
```
[在此貼上你的 API 程式碼或描述，例如：
POST /api/v1/users
功能：建立新用戶
認證：Bearer Token
]
```

輸出格式（Markdown）：

## [HTTP Method] [Endpoint]
> 一句話描述

### Request
| 欄位 | 類型 | 必填 | 說明 |
|------|------|------|------|

### Response
**成功回應（200）：**
```json
[範例 JSON]
```

**錯誤回應：**
| Status Code | 錯誤訊息 | 說明 |

### 程式碼範例
- cURL
- JavaScript (fetch)
- Python (requests)

### 注意事項（若有）
```

## Prompt 20：Debug 分析助手

**情境**：遇到 bug 不知道從何下手。

**Prompt**：
```
你是一位擅長系統性 Debug 的工程師，使用「假設 → 驗證」的科學方法。

問題描述：[填寫，越詳細越好]
錯誤訊息：[填寫，貼上完整 stack trace]
環境：[填寫，例：Node.js 22, macOS, Apple Silicon]
最近的改動：[填寫，例：昨天更新了 npm package X 後開始出現]

程式碼（相關部分）：
```[語言]
[貼上相關程式碼]
```

請提供：

1. **問題診斷**：最可能的根本原因（Top 3，按可能性排序）

2. **驗證步驟**：針對每個假設，提供 1-2 個可以快速驗證的指令或測試

3. **解決方案**：
   - 快速解（暫時 workaround）
   - 根本解（根治問題的方式）

4. **預防措施**：如何避免同類問題再次發生

請使用繁體中文解釋，程式碼部分保持英文。
```

## Prompt 21：架構設計 Review

**情境**：設計一個新系統架構，需要獨立審查。

**Prompt**：
```
你是一位系統架構師，曾設計過每日百萬用戶的大型系統。

我的系統需求：
- 功能：[填寫]
- 預期流量：[填寫，例：初期 1K MAU，一年後 50K MAU]
- 技術棧：[填寫]
- 預算限制：[填寫，例：月費控制在 $100 以內]

我目前的架構設計：
[填寫或貼上架構圖描述 / 技術選型清單]

請從以下維度 Review：

1. **可擴展性**：目前架構能撐到什麼量級？瓶頸在哪裡？
2. **可維護性**：是否有過度設計或欠缺設計的地方？
3. **安全性**：明顯的安全隱患？
4. **成本效益**：是否有更省錢的替代方案？
5. **台灣合規**：若涉及個人資料，是否符合台灣個資法要求？

給出：整體評分（1-10）+ 最重要的 3 個改善建議（排序）
```

## Prompt 22：台灣支付整合（綠界 / 藍新）

**情境**：需要在台灣電商網站整合金流，不確定如何設計。

**Prompt**：
```
你是一位有豐富台灣電商開發經驗的工程師，熟悉綠界科技（ECPay）和藍新金流（NewebPay）的 API。

我的需求：
- 框架/語言：[填寫，例：Next.js 15 / Node.js]
- 需要的支付方式：[填寫，例：信用卡 + ATM 轉帳 + 超商代碼]
- 訂單類型：[填寫，例：一次性購買 / 定期扣款]
- 特殊需求：[填寫，例：需要開發票 / 需要退款功能]

請提供：

1. **綠界 vs 藍新比較**（針對我的需求）
   - 費率比較
   - API 整合難度
   - 適合場景

2. **整合架構設計**
   - Webhook 處理流程（訂單狀態同步）
   - 防重複付款機制
   - 退款流程

3. **核心程式碼片段**（以 [填寫框架] 為例）
   - 建立訂單
   - Webhook 接收與驗證
   - 查詢付款狀態

4. **常見地雷**：台灣金流整合最容易踩的 3 個坑

使用繁體中文說明，程式碼加繁中注解。
```

## Prompt 23：Git Commit Message 規範

**情境**：想為團隊建立一致的 Git commit 規範。

**Prompt**：
```
你是一位 DevOps 工程師，熟悉 Conventional Commits 規範和語意化版本控制。

我的專案類型：[填寫，例：SaaS Web App / CLI 工具 / npm 套件]
團隊規模：[填寫，例：5 人]
現有問題：[填寫，例：commit 訊息格式不一致，很難追蹤變更原因]

請設計一套適合我的 Git Commit 規範：

1. **Type 定義**（含中文說明與使用時機）
   - feat / fix / docs / style / refactor / test / chore 等

2. **Commit Message 模板**（附範例 10 條，涵蓋常見場景）

3. **Commitlint 設定**（package.json / .commitlintrc.js 範例）

4. **Husky pre-commit hook 設定**（自動檢查 + lint）

5. **台灣工程師常犯的 Commit 錯誤**（3 個壞習慣 + 改善方法）

輸出完整可複製的設定檔，附繁中說明。
```
